Use Case Note
■ Workflow
    • // Write down briefly how user perform the work
    • El usuario accede al módulo de facturación.
    • Busca y selecciona la factura a anular.
    • El sistema valida condiciones para anulación.
    • El usuario confirma la anulación.
    • El sistema marca la factura como anulada.
    • Se evita ajustar inventario físicamente para evitar descuadres.
    • Se actualiza historial y registros contables.
    • Se notifica la operación al usuario.
■ Business Logic
    • // Write down what user expect the system to react upon certain condition (e.g. low inventory alert level)
    • Solo facturas en periodo abierto y no anuladas previamente pueden ser anuladas.
    • La anulación debe dejar registro histórico inalterable.
    • El inventario no se modifica físicamente para evitar inconsistencias.
    • El sistema debe actualizar registros contables vinculados.
    • Se deben mantener trazabilidad y auditoría claras y detalladas.
    • Se deben mostrar mensajes claros en caso de restricciones o errores.

Scenario
Steps
1. El ADMINISTRADOR accede al módulo de facturación.
2. Solicita la anulación de una factura existente mediante búsqueda por número o cliente.
3. El sistema valida que la factura exista y esté anulable según política (no estar ya anulada, periodo abierto, etc.).
4. El sistema muestra el detalle de la factura y la orden asociada.
5. El ADMINISTRADOR confirma la anulación.
6. El sistema registra la anulación de la factura, marcándola como anulada en la base de datos.
7. El sistema revisa el impacto en inventario:
    7.1. No realiza ajustes físicos en inventario para evitar descuadres.
    7.2. Registra el movimiento contable y administrativo correspondiente.
8. El sistema actualiza el historial de la orden para reflejar la anulación.
9. El sistema notifica al usuario la anulación exitosa.
10. El caso de uso finaliza.
