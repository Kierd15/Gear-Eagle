Use Case Note
■ Workflow
    • // Write down briefly how user perform the work
    • El sistema detecta un evento que requiere notificación (aprobación, rechazo, cambio de estado, recordatorio, etc.).
    • El sistema identifica el destinatario correspondiente según reglas de asignación (mecánico, supervisor, cliente, administrador).
    • El sistema genera el mensaje con la información relevante del evento.
    • Se envía la notificación a través del canal configurado (app, correo electrónico, SMS, etc.).
    • El sistema registra el envío y el estado de la notificación (enviada, entregada, leída).
    • Si aplica, el destinatario puede confirmar la recepción o realizar una acción desde la notificación.
■ Business Logic
    • // Write down what user expect the system to react upon certain condition (e.g. low inventory alert level)
    • Solo se deben generar notificaciones ante eventos válidos y previamente confirmados por el sistema.
    • El contenido de la notificación debe incluir información mínima obligatoria (ID del caso/orden, tipo de evento, fecha y acción requerida).
    • El sistema debe validar que el destinatario tenga permisos y esté activo antes de enviar la notificación.
    • Todas las notificaciones deben quedar registradas para fines de auditoría (quién, cuándo, qué evento y resultado del envío).
    • En caso de falla en el envío, el sistema debe ejecutar reintentos automáticos según política definida.
    • Si la notificación no es atendida dentro de un tiempo establecido, el sistema debe escalarla al siguiente nivel jerárquico.
    • El sistema debe evitar el envío de notificaciones duplicadas para el mismo evento.

Scenario
Steps
1. Escenario A: Notificación al cliente (moto lista):
    1.1. MECANICO actualiza el estado de la moto a “lista para entrega”.
    1.2. Sistema registra el cambio de estado y prepara notificación.
    1.3. Sistema selecciona método de notificación preferido del cliente (correo, SMS).
    1.4. Sistema envía notificación:
        1.4.1. if envío exitoso:
            1.4.1.1. registra fecha y hora de envío.
               end if
        1.4.2. if fallo en envío:
            1.4.2.1. intenta método alternativo (correo si falló SMS, SMS si falló correo).
               end if
    1.5. CLIENTE recibe notificación.
    1.6. CLIENTE puede confirmar recepción o programar recogida desde la plataforma.
    1.7. MECANICO recibe confirmación de que el cliente fue notificado.
2. Escenario B: Notificación de stock mínimo al administrador
    2.1. Sistema monitorea inventario de todos los productos.
    2.2. Detecta que algún producto ha alcanzado el stock mínimo definido para ese artículo.
    2.3. Sistema genera notificación al administrador:
        2.3.1. Incluye nombre del producto, cantidad disponible y stock mínimo definido.
    2.4. Administrador recibe notificación vía app, correo o dashboard.
    2.5. Administrador revisa información y decide reabastecer:
        2.5.1. if realiza pedido:
            2.5.1.1. sistema registra acción y alerta se cierra cuando stock actualizado.
               end if
        2.5.2. while stock < mínimo:
            2.5.2.1. alerta permanece activa hasta que se complete reabastecimiento.
               end while
    2.6. Sistema registra historial de alertas para auditoría y análisis.
